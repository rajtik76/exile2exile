# Architecture

A high-level map of the codebase. For the game-data release pipeline (patch
detection, staging, CI validation, atomic activation) see the "Game data"
section of the [README](../README.md); this document covers the application
itself.

## Big picture

The app is a Laravel 13 backend with an Inertia v3 / React 19 frontend - a
server-routed SPA, no separate API layer. There is no user authentication:
planning, importing and sharing builds are guest actions, keyed by slugs and
edit tokens instead of accounts.

All PoE2 game data the app serves (passive tree, items, gems, runes, icons) is
read from an extracted GGPK release behind the `current` symlink; nothing is
committed to the repository. The extraction itself lives in
[poe2-toolkit](https://github.com/rajtik76/poe2-toolkit) and is driven from
`tools/poe-data-extract`.

## Backend (`app/`)

| Namespace | Responsibility |
|---|---|
| `Pob/` | Path of Building import: decoding export codes, build sources (raw code, pobb.in), snapshot data objects, validation, icon and mod resolution against the GGPK extract |
| `Support/Planner/` | The build planner document: plan schema, references, mapping a decoded PoB snapshot into a plan |
| `Build/` | Passive-tree indexing and the shared build document assembled for the tree viewer |
| `Filter/` | Loot-filter generation: a typed grammar of conditions, actions and blocks, NeverSink base filters, economy- and build-driven builders |
| `Economy/` | Market prices from poe2scout: client, price book, priced items |
| `Services/`, `Jobs/`, `Console/` | Game-data release store (staging, atomic swap, pruning), the patch watcher, staging and CI-trigger jobs, deploy commands |
| `Http/` | Controllers, form requests and middleware; form requests carry domain validation (an import request resolves and validates the PoB code, then stashes the snapshot for the controller) |
| `Models/` | Eloquent models: build plans, shared builds, patch releases and subscribers, economy prices, page views |
| `Support/` | Small shared value objects and helpers (versions, hashes, patch status) |

Conventions worth knowing:

- Domain types are final, mostly readonly value objects; enums render their own
  filter syntax (see `Filter/`). Factories like `Filter\Conditions` are the only
  way to build filter lines, so a filter cannot be assembled from mistyped
  keywords.
- Validation lives in form requests, not controllers. Requests that resolve
  external input (a PoB code) do the resolution during validation and hand the
  controller a ready snapshot.
- Everything that reads game data goes through the `current` release symlink
  paths (`public/tree/current`, `public/icons/poe2`, `resources/poe2/ggpk`),
  wired by `php artisan poe2:link-game-data`.

## Frontend (`resources/js/`)

- `pages/` - Inertia pages: the welcome page, the passive-tree viewer, the
  build planner and the small static pages (changelog, stats, legal).
- `components/` - `passive-tree/` (canvas rendering), `planner/`, `build/`
  (shared build blocks such as the tooltip system) and `ui/` (shadcn/ui).
- `lib/` - pure TypeScript modules with 1:1 vitest coverage: tree scene and
  sprite-atlas handling, planner state and plan document logic, mod-line
  parsing, guards. Logic goes here, components stay thin.
- `actions/` and `routes/` are generated by Laravel Wayfinder and gitignored;
  they regenerate on build.

The passive tree renders to canvas from GGPK sprite atlases; tree data and
atlases are fetched from the versioned release the backend serves.

## Testing

Four suites with different contracts (see "Running tests" in the README):

- `tests/Unit` - pure Pest tests, no Laravel bootstrapping.
- `tests/Feature` - HTTP and service tests; every external process and the
  game-data disk are faked.
- `tests/Contract` - runs against a real extracted release, guarding the
  app <-> data contract (tree shape, icons present, PoB import end to end).
  CI runs it on the exact artifact production will serve.
- `tests/Browser` - Pest browser tests through Playwright.

JS logic in `resources/js/lib` is covered by vitest. Quality gates: pint,
phpstan (level 7), rector, eslint, prettier and tsc, wired into the pre-commit
hook and CI (`composer review` / `composer ci:check`).
